# init-db

This bootstrap service is intentionally one-shot.

## Schema-per-service ownership model

Each application schema is created and owned by the Postgres superuser identified by `POSTGRES_USER`. Service roles are granted access to their schema; they do not own the schema and they do not run DDL.

This split is deliberate:

- Migrations run as the superuser so schema changes can be applied without granting each service role broader DDL capabilities.
- Application roles only need data-plane access for normal operation.
- Keeping ownership centralized avoids privilege escalation paths where a service account could change schema objects or grant itself additional access.

In practice, the bootstrap creates a schema per service and grants each service role the privileges it needs for DML, while ownership remains with the migration runner.

## Why `ALTER DEFAULT PRIVILEGES` is needed

`GRANT ... ON ALL TABLES IN SCHEMA ...` is a point-in-time grant. It applies only to tables that already exist when the command runs during bootstrap.

That means a later migration that creates a new table would not automatically make that table visible to the service role. Without an additional step, developers would need to re-run manual grants after every migration that introduces new tables or sequences.

`ALTER DEFAULT PRIVILEGES FOR ROLE <superuser> IN SCHEMA <schema>` fixes that. It instructs Postgres to automatically apply the configured grants to every future table or sequence created by that role in that schema.

Because migrations run as `POSTGRES_USER`, the default privilege rule must be scoped with `FOR ROLE <superuser>` matching that migration runner role.

## The `public` schema

No application tables are created in `public`.

The `public` schema is reserved for Postgres extensions only, such as `pgcrypto`. Application code must always use schema-qualified names when referring to tables. For example, use `secrets.secret_entries`, not `secret_entries`.

This keeps application data isolated by service boundary and avoids ambiguous lookups through the default search path.

## How to verify grants after bootstrap

To inspect the configured default privileges for future tables and sequences, run:

```sql
SELECT n.nspname AS schema,
	   r.rolname AS defaclrole,
	   pg_catalog.array_to_string(d.defaclacl, ', ') AS acl
FROM pg_default_acl d
JOIN pg_namespace n ON n.oid = d.defaclnamespace
JOIN pg_roles r ON r.oid = d.defaclrole
ORDER BY schema;
```

To inspect current table grants that already exist in the database, run:

```sql
SELECT grantee, table_schema, privilege_type
FROM information_schema.role_table_grants
WHERE table_schema NOT IN ('pg_catalog', 'information_schema')
ORDER BY table_schema, grantee;
```

Together, these queries show both sides of the privilege model:

- The current grants on tables that already exist.
- The default grants that will be applied to future objects created by migrations.

To prove the defaults work at runtime for migration-created objects, run an end-to-end check after bootstrap:

1. Connect as the migration runner role (`POSTGRES_USER`) and create a test table in the service schema you want to verify. Use an identity column so sequence privileges are exercised too.

```sql
CREATE TABLE goal_engine.bootstrap_acl_runtime_check (
	id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
	note text NOT NULL
);
```

2. Open a new session as the corresponding service role (`lifeos_goal_engine_rw` for `goal_engine`) and perform write and read operations against that newly created table.

```sql
INSERT INTO goal_engine.bootstrap_acl_runtime_check (note)
VALUES ('runtime validation')
RETURNING id, note;

SELECT id, note
FROM goal_engine.bootstrap_acl_runtime_check;

UPDATE goal_engine.bootstrap_acl_runtime_check
SET note = 'runtime validation updated'
WHERE note = 'runtime validation';

DELETE FROM goal_engine.bootstrap_acl_runtime_check
WHERE note = 'runtime validation updated';
```

3. Reconnect as the migration runner role and clean up the test object.

```sql
DROP TABLE goal_engine.bootstrap_acl_runtime_check;
```

This is the post-bootstrap proof that both migration-time default privileges and runtime service-role access are working correctly for newly created tables and their backing sequences.

## Bootstrap ordering contract

The `init-db` service runs on `image: postgres:16` (Debian-based) because `scripts/init-postgres.sh` requires bash (`#!/usr/bin/env bash`, `set -euo pipefail`, and bash arrays). Do not use `postgres:16-alpine` for this service because Alpine ships BusyBox `sh` and does not provide bash by default.

Startup ordering contract:

1. `postgres` (image: `postgres:16-alpine`) must reach `service_healthy` (`pg_isready` passes).
2. `init-db` (image: `postgres:16`) runs `scripts/init-postgres.sh` and must exit `0`.
3. All DB-dependent app services start only after `init-db` reaches `service_completed_successfully`.

This uses a one-shot bootstrap pattern and remains safe to re-run because schema creation, role creation, grants, and default privilege configuration are idempotent.

## Diagnosing bootstrap failures

When `init-db` fails, run these checks in order:

1. Check the container exit code and logs:

```bash
docker compose logs init-db
```

2. Verify the image in use is `postgres:16` (not `postgres:16-alpine`):

```bash
docker compose config --format json | grep -A2 '"init-db"'
```

3. Confirm bash is available inside the container:

```bash
docker compose run --rm init-db bash --version
```

4. Confirm required env vars are set (the script uses `:?` guards that print the missing variable name on failure):

```bash
docker compose run --rm init-db env | grep POSTGRES
```

5. Confirm the postgres service is healthy before re-running init-db:

```bash
docker compose ps postgres
```
